# Swagger UI 테스트

1. 서버를 실행한다.
2. 브라우저에서 `http://127.0.0.1:8000/docs`에 접속한다.
3. 테스트할 API를 펼친다.
4. `Try it out`을 클릭한다.
5. 요청 값을 입력한다.
6. `Execute`를 클릭한다.
7. Response body와 status code를 확인한다.

## Swagger UI와 Postman 차이

Swagger UI와 Postman은 모두 API를 테스트하는 도구입니다. 다만 사용하는 상황이 조금 다릅니다.

| 구분 | Swagger UI | Postman |
| --- | --- | --- |
| 실행 방식 | FastAPI 서버를 실행하면 `/docs`에서 자동으로 열립니다. | 별도 프로그램을 설치해서 사용합니다. |
| 준비 난이도 | 설치가 필요 없어 초보자가 바로 사용하기 쉽습니다. | 설치와 기본 사용법을 먼저 익혀야 합니다. |
| API 목록 확인 | FastAPI 코드 기준으로 API 목록과 요청 모델이 자동 표시됩니다. | 직접 URL, Method, Body를 입력하거나 Collection을 만들어야 합니다. |
| 요청 테스트 | `Try it out`으로 빠르게 테스트할 수 있습니다. | 여러 요청을 저장하고 반복 테스트하기 좋습니다. |
| 협업/문서화 | 현재 서버의 API 구조를 확인하는 데 좋습니다. | 팀 단위 API 모음, 환경 변수, 테스트 케이스 관리에 좋습니다. |
| 수업에서의 역할 | FastAPI를 처음 배울 때 기본 테스트 도구로 사용합니다. | API 테스트를 더 체계적으로 저장하고 반복할 때 선택적으로 사용합니다. |

## 장단점

Swagger UI의 장점:

- FastAPI가 자동으로 만들어 주므로 별도 설치가 필요 없습니다.
- 요청 Body 예시와 Pydantic 검증 구조를 바로 확인할 수 있습니다.
- 초보자가 API 주소, Method, status code를 처음 익히기에 좋습니다.

Swagger UI의 단점:

- 테스트 요청을 프로젝트별로 저장하고 관리하기에는 제한적입니다.
- 여러 환경의 API 주소를 바꿔 가며 테스트하기에는 불편할 수 있습니다.
- 팀 협업용 API Collection 관리에는 Postman이 더 적합합니다.

Postman의 장점:

- 요청을 Collection으로 저장해 반복 테스트할 수 있습니다.
- 개발, 테스트, 배포 환경별 URL과 token을 변수로 관리하기 좋습니다.
- 팀 프로젝트에서 API 테스트 결과를 공유하기 쉽습니다.

Postman의 단점:

- 별도 설치가 필요합니다.
- 초보자는 Method, URL, Header, Body 설정을 직접 해야 해서 처음에는 헷갈릴 수 있습니다.
- FastAPI 코드에서 자동 생성되는 문서와 항상 동기화되는 것은 아니므로, API가 바뀌면 요청도 직접 수정해야 합니다.

## 수업 기준

처음에는 Swagger UI로 API 구조와 요청/응답을 확인합니다. 이후 같은 API를 반복 테스트하거나 팀 프로젝트에서 요청 목록을 저장해야 할 때 Postman을 선택적으로 사용합니다.
